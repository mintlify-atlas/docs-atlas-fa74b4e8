Active Job is a framework for declaring background jobs and making them run on a variety of queuing backends. It provides a standardized interface so jobs can be everything from scheduled clean-ups and billing charges to mailings—anything that can be chopped into small units of work and run in parallel.

By offloading time-consuming operations from the main request-response cycle, Active Job keeps your application responsive while tasks execute in the background.

## Creating a Job

<Steps>
  <Step title="Generate the job">
    Use the Rails generator to create a job file and its corresponding test:

    ```bash
    $ bin/rails generate job guests_cleanup
    invoke  test_unit
    create    test/jobs/guests_cleanup_job_test.rb
    create  app/jobs/guests_cleanup_job.rb
    ```

    To generate a job pinned to a specific queue:

    ```bash
    $ bin/rails generate job guests_cleanup --queue urgent
    ```
  </Step>

  <Step title="Define the perform method">
    Jobs live in `app/jobs/` and inherit from `ApplicationJob`. The `perform` method contains the work to be done:

    ```ruby
    # app/jobs/guests_cleanup_job.rb
    class GuestsCleanupJob < ApplicationJob
      queue_as :default

      def perform(*guests)
        guests.each(&:destroy)
      end
    end
    ```

    `perform` accepts any number of arguments. Active Job serializes them automatically for storage in the queue.
  </Step>

  <Step title="Enqueue the job">
    Call `perform_later` to enqueue the job for background execution:

    ```ruby
    # Enqueue immediately—runs when the queue is free
    GuestsCleanupJob.perform_later(guest)

    # Enqueue to run tomorrow at noon
    GuestsCleanupJob.set(wait_until: Date.tomorrow.noon).perform_later(guest)

    # Enqueue to run in 1 week
    GuestsCleanupJob.set(wait: 1.week).perform_later(guest)

    # Pass multiple arguments
    GuestsCleanupJob.perform_later(guest1, guest2, filter: "inactive")
    ```
  </Step>
</Steps>

## Running Jobs Synchronously

Use `perform_now` when you need the job to run inline, in the same process:

```ruby
GuestsCleanupJob.perform_now(guest)
```

<Note>
  `perform_now` bypasses the queue entirely—useful for testing or simple scripts, but blocks the current thread until the job finishes.
</Note>

## Queues and Priority

### Assigning a Queue

```ruby
class GuestsCleanupJob < ApplicationJob
  queue_as :low_priority
end
```

Override the queue at enqueue time:

```ruby
MyJob.set(queue: :another_queue).perform_later(record)
```

Prefix all queues for an environment in `config/application.rb`:

```ruby
config.active_job.queue_name_prefix = Rails.env
# Results in: production_low_priority, staging_low_priority, etc.
```

### Dynamic Queue Selection

Pass a block to `queue_as` to choose the queue based on job arguments:

```ruby
class ProcessVideoJob < ApplicationJob
  queue_as do
    video = self.arguments.first
    video.owner.premium? ? :premium_videojobs : :videojobs
  end

  def perform(video)
    # process video
  end
end
```

### Job Priority

Assign a numeric priority (lower numbers = higher priority within a queue):

```ruby
class GuestsCleanupJob < ApplicationJob
  queue_with_priority 10
end

# Or at enqueue time:
MyJob.set(priority: 50).perform_later(record)
```

<Info>
  With Solid Queue (the default backend), queue order takes precedence over priority numbers. Priority only applies within a single queue.
</Info>

## Default Backend: Solid Queue

Solid Queue is a database-backed queuing system included by default from Rails 8.0. It requires no external dependencies like Redis.

<Tabs>
  <Tab title="Development Setup">
    Add to `config/environments/development.rb`:

    ```ruby
    config.active_job.queue_adapter = :solid_queue
    config.solid_queue.connects_to = { database: { writing: :queue } }
    ```

    Add a `queue` entry to `config/database.yml`:

    ```yaml
    development:
      primary:
        <<: *default
        database: storage/development.sqlite3
      queue:
        <<: *default
        database: storage/development_queue.sqlite3
        migrations_paths: db/queue_migrate
    ```

    Prepare the database and start the queue:

    ```bash
    $ bin/rails db:prepare
    $ bin/jobs start
    ```
  </Tab>
  <Tab title="Production Setup">
    Solid Queue is pre-configured in new Rails 8 apps:

    ```ruby
    # config/environments/production.rb
    config.active_job.queue_adapter = :solid_queue
    config.solid_queue.connects_to = { database: { writing: :queue } }
    ```

    ```yaml
    # config/database.yml
    production:
      primary:
        <<: *default
        database: storage/production.sqlite3
      queue:
        <<: *default
        database: storage/production_queue.sqlite3
        migrations_paths: db/queue_migrate
    ```
  </Tab>
  <Tab title="Queue Configuration">
    Configure workers and dispatchers in `config/queue.yml`:

    ```yaml
    default: &default
      dispatchers:
        - polling_interval: 1
          batch_size: 500
      workers:
        - queues: "*"
          threads: 3
          processes: <%= ENV.fetch("JOB_CONCURRENCY", 1) %>
          polling_interval: 0.1
    ```
  </Tab>
</Tabs>

## Alternate Queuing Backends

You can swap Solid Queue for any supported adapter without changing your job code:

```ruby
# config/application.rb
config.active_job.queue_adapter = :sidekiq

# Or per-job:
class GuestsCleanupJob < ApplicationJob
  self.queue_adapter = :resque
end
```

Popular adapters include Sidekiq, Resque, Delayed Job, Queue Classic, Sneakers, and Que. See the [ActiveJob::QueueAdapters API docs](https://api.rubyonrails.org/classes/ActiveJob/QueueAdapters.html) for the full list.

## Retrying and Discarding Failed Jobs

By default, a failed job is not retried. Use `retry_on` and `discard_on` to configure failure handling:

```ruby
class RemoteServiceJob < ApplicationJob
  # Retry up to 5 times with 3-second wait (defaults)
  retry_on CustomAppException

  # Retry with custom options
  retry_on NetworkTimeoutError, wait: :polynomially_longer, attempts: 10

  # Silently discard the job on this error
  discard_on ActiveJob::DeserializationError

  def perform(*args)
    # work that might raise errors
  end
end
```

Handle exceptions inline with `rescue_from`:

```ruby
class GuestsCleanupJob < ApplicationJob
  rescue_from(ActiveRecord::RecordNotFound) do |exception|
    # log or notify
  end

  def perform
    # ...
  end
end
```

## Callbacks

Active Job provides hooks for the full job lifecycle:

```ruby
class GuestsCleanupJob < ApplicationJob
  queue_as :default

  around_perform :around_cleanup

  before_enqueue { |job| Rails.logger.info "Enqueueing #{job.class}" }
  after_perform  { |job| Rails.logger.info "Finished #{job.class}" }

  def perform
    # do work
  end

  private
    def around_cleanup
      # before perform
      yield
      # after perform
    end
end
```

### Available Callbacks

<Accordion title="Enqueue callbacks">
  - `before_enqueue` — runs before the job is placed on the queue
  - `around_enqueue` — wraps the enqueue call
  - `after_enqueue` — runs after the job is successfully enqueued
</Accordion>

<Accordion title="Perform callbacks">
  - `before_perform` — runs before `perform` is called
  - `around_perform` — wraps the `perform` call
  - `after_perform` — runs after `perform` completes
  - `after_discard` — runs when a job is discarded via `discard_on`
</Accordion>

<Note>
  When using `perform_all_later` for bulk enqueuing, per-job enqueue callbacks (`around_enqueue`, etc.) are not triggered on individual jobs.
</Note>

## Bulk Enqueuing

Enqueue multiple jobs in a single round-trip to the queue backend:

```ruby
# Create job instances
cleanup_jobs = Guest.all.map { |guest| GuestsCleanupJob.new(guest) }

# Enqueue all at once
ActiveJob.perform_all_later(cleanup_jobs)

# Can mix different job classes
ActiveJob.perform_all_later(cleanup_job, export_job, notify_job)
```

## GlobalID Support

Active Job automatically serializes Active Record objects via GlobalID—no manual class/id pairs needed:

```ruby
# Without GlobalID (old approach)
class TrashableCleanupJob < ApplicationJob
  def perform(trashable_class, trashable_id, depth)
    trashable = trashable_class.constantize.find(trashable_id)
    trashable.cleanup(depth)
  end
end

# With GlobalID (current approach)
class TrashableCleanupJob < ApplicationJob
  def perform(trashable, depth)
    trashable.cleanup(depth)
  end
end
```

<Warning>
  If a record is deleted after the job is enqueued but before `perform` runs, Active Job raises `ActiveJob::DeserializationError`. Use `discard_on ActiveJob::DeserializationError` to handle this gracefully.
</Warning>

## Recurring Tasks

Solid Queue supports cron-style recurring tasks via `config/recurring.yml`:

```yaml
production:
  daily_cleanup:
    class: GuestsCleanupJob
    schedule: every day at 9am
  periodic_report:
    class: MyJob
    args: [42, { status: "custom_status" }]
    schedule: every second
```

## Integration with Action Mailer

Active Job powers `deliver_later` in Action Mailer:

```ruby
# Synchronous—blocks until sent
UserMailer.welcome(@user).deliver_now

# Asynchronous—enqueues via Active Job
UserMailer.welcome(@user).deliver_later
```
